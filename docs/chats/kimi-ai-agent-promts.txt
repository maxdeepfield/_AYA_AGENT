# 🧬 Self-Evolving AI Agent

Самоэволюционирующий AI-агент, который улучшает свой собственный код через цикл генерации, критики и отбора.

## 🎯 Концепция

Это **минимальная психика** — система из нескольких ролей (LLM), которые работают вместе:

1. **GENERATOR** (Мечтатель) — предлагает изменения кода
2. **CRITIC** (Скептик) — анализирует и даёт штрафы
3. **JUDGE** (Судья) — принимает решение на основе fitness
4. **REALITY** (Реальность) — запускает код и проверяет что он работает
5. **HISTORIAN** (Историк) — помнит прошлые попытки

## 🧮 Fitness Formula

```typescript
fitness = 
  + 50 * alive                    // Агент не упал
  + 10 * behaviorChanged          // Поведение изменилось
  -  1 * (tokenEstimate / 1000)   // Штраф за токены
  -  1 * (|codeSizeDelta| / 5000) // Штраф за рост кода
  + criticPenalty                 // Штрафы от критика (-5 до 0)
```

**Порог принятия:** `fitness >= 5`

## 🎭 Роли в деталях

### GENERATOR (Мечтатель)

**Задача:** Предложить улучшение кода

**Входы:**
- Текущий код `agent.ts`
- История последних 5 попыток (успешные/отклонённые)

**Выход:**
- `pattern`: краткое описание изменения
- `code`: новый код агента

**Промпт:**
- Может использовать HTTP, shell, любые возможности
- Не должен повторять отклонённые паттерны из истории
- Должен делать маленькие изменения
- Три типа мутаций: EXPLORATORY / STABILIZATION / REFINEMENT

### CRITIC (Скептик)

**Задача:** Найти проблемы в патче

**Входы:**
- Старый код
- Новый код

**Выход:**
- `penalties`: штрафы от -5 до 0
- `warnings`: предупреждения (не влияют на fitness)

**Только 3 причины для penalties:**
1. `capability_claim` (-5): заявляет способности которых нет
2. `undefined_behavior` (-3): неопределённое поведение
3. `removes_behavior` (-2): удаляет существующее поведение

**Важно:** CRITIC не имеет права veto! Он только даёт штрафы, решение принимает JUDGE.

### REALITY (Реальность)

**Задача:** Запустить код и проверить что он работает

**Процесс:**
1. Запускает baseline (старый код) на 3 секунды
2. Запускает candidate (новый код) на 3 секунды
3. Собирает все messages от агента
4. Проверяет: упал или нет

**Выход:**
- `alive`: true/false
- `messages`: массив сообщений от агента
- `error`: текст ошибки если упал

### SCORER

**Задача:** Посчитать fitness

**Логика:**
- Если агент упал → `alive = false` → fitness = -100 (автоматический reject)
- Если поведение изменилось (messages разные) → +10
- Штраф за токены и размер кода
- Добавляет penalties от критика

### JUDGE (Судья)

**Задача:** Принять решение

**Логика:**
```typescript
if (fitness >= 5) {
  ACCEPT → сохранить новый код
} else {
  REJECT → откатить к старому коду
}
```

### HISTORIAN (Историк)

**Задача:** Помнить прошлое

**Хранит в `state.json`:**
- Последние 20 попыток
- Для каждой: accepted/rejected, pattern, fitness, scores, signals

**GENERATOR видит:**
- Последние 5 попыток
- Список отклонённых паттернов с причинами
- Список успешных паттернов

## 🔄 Цикл эволюции

```
1. Читаем agent.ts
2. GENERATOR предлагает патч
3. CRITIC анализирует → penalties
4. REALITY запускает baseline и candidate
5. SCORER считает fitness
6. JUDGE решает: ACCEPT или REJECT
7. HISTORIAN сохраняет результат
8. Ждём 5 секунд
9. Повторяем
```

## 🧠 Философия

### 1. Не навязывать протоколы

Агент **не обязан** следовать какому-то event-протоколу. Он сам решает как показать свои способности.

### 2. CRITIC без veto

CRITIC не может заблокировать патч. Он только даёт штрафы, а решение принимает JUDGE на основе fitness.

### 3. Измерять реальное поведение

Не верим на слово — запускаем код и смотрим что происходит.

### 4. Память паттернов

GENERATOR видит историю и не повторяет отклонённые паттерны.

### 5. Минимальная психика

Система из нескольких ролей создаёт **конфликт** → из конфликта рождается интеллект.

## 🚀 Что дальше?

Возможные направления развития:

1. **COMPACT режим** — компрессия кода при превышении бюджета
2. **Task planner** — агент сам ставит себе задачи
3. **Capability registry** — явное отслеживание способностей
4. **Multi-agent** — несколько агентов эволюционируют параллельно
5. **Genetic crossover** — скрещивание успешных паттернов

## 🤔 Почему это работает?

1. **Конфликт ролей** — GENERATOR мечтает, CRITIC ломает, JUDGE решает
2. **Память** — не повторяем ошибки
3. **Реальность** — код должен работать, не только компилироваться
4. **Давление отбора** — fitness формула создаёт направление эволюции

## ⚠️ Ограничения

- Агент имеет root-доступ (может делать всё)
- Нет песочницы (запускается на реальной машине)
- Нет лимита процессов
- Нет проверки безопасности

**Это эксперимент, не production система!**

1. Ниша: Максимальная стабильность (самая простая)
Цель: Выжить как можно дольше, минимизировать падения.
TypeScript
Copy
// REALITY собирает:
const stats = {
  uptime: process.uptime(),           // Секунд с запуска
  crashCount: loadCrashHistory(),     // Количество падений за всё время
  memoryLeak: memoryUsage.rss - baselineMemory
}

// SCORER:
fitness = 
  + stats.uptime * 10                // Чем дольше живёт, тем лучше
  - stats.crashCount * 50            // Каждый краш — тяжёлый штраф
  - stats.memoryLeak / 1024 / 1024   // Штраф за утечку памяти (МБ)
  - codeSizeDelta / 1000             // Лёгкость тоже ценится
Эффект: Агент будет удалять рисковый код, упрощаться, становиться "резиновым" — развитие в сторону надёжности.
2. Ниша: Максимальная информативность (мой фаворит)
Цель: Генерировать как можно больше уникальной информации о своём состоянии.
TypeScript
Copy
// REALITY собирает:
const signals = {
  messageEntropy: calculateEntropy(messages),  // Уникальность паттернов
  stateVariance: countUniqueStateChanges(),    // Разнообразие внутренних состояний
  cyclicDepth: detectCycleDepth(messages)      // Глубина циклических структур
}

// SCORER:
fitness = 
  + signals.messageEntropy * 20      // Научись говорить по-разному
  + signals.stateVariance * 15       // Меняй внутреннее состояние
  + signals.cyclicDepth * 5          // Порождай сложные паттерны
  - (idleTime * 10)                  // Штраф за бездействие
Эффект: Агент начнёт создавать внутренние переменные, циклы, накапливать историю, экспериментировать с форматом вывода — эмерджентная сложность.

🎯 Практический совет: Начни с #2 (информативность)
Почему:
Легко измерить (entropy = count unique patterns)
Не требует внешних действий
Поощряет эксперименты, а не консервацию
Ты увидишь рост сложности на глазах

TypeScript
Copy
function calculateEntropy(messages: string[]): number {
  const patterns = messages.map(m => m.replace(/\d+/g, '#')); // Нормализуем числа
  const frequencies = patterns.reduce((acc, p) => {
    acc[p] = (acc[p] || 0) + 1;
    return acc;
  }, {} as Record<string, number>);
  
  const total = patterns.length;
  return Object.values(frequencies).reduce((sum, freq) => {
    const p = freq / total;
    return sum - p * Math.log2(p);
  }, 0);
}
Что ты увидишь:
Поколение 1-5: "I am alive. I am alive. I am alive." (entropy ~0)
Поколение 10-20: Добавление счётчиков, времени, случайных чисел
Поколение 30+: Структурированные JSON-отчёты, вложенные циклы, саморефлексия
Главное правило для "вакуума": Задача должна быть измерима внутри системы и негативно штрафовать отсутствие действия. Иначе агент найдёт оптимум в "лежании на диване".

## Сценарий 1: "Один агент, много циклов"

```
Цикл 1: Agent → fitness=10 → ACCEPT
Цикл 2: Agent → fitness=15 → ACCEPT
Цикл 3: Agent → fitness=8  → REJECT
Цикл 4: Agent → fitness=20 → ACCEPT
...
Цикл 100: Agent → fitness=45 → ACCEPT
```

**Результат:** Один агент, который постепенно улучшается.

## Сценарий 2: "Популяция, турнир"

```
Цикл 1-50:
  Agent A → fitness=10, 15, 8, 20... → avg=15
  Agent B → fitness=5, 12, 18, 9... → avg=11
  Agent C → fitness=8, 7, 6, 5... → avg=6.5

Турнир: A выживает, B выживает, C удаляется
Рождаются новые мутанты от A и B

Цикл 51-100:
  Agent A → fitness=20, 25, 22...
  Agent B → fitness=18, 20, 19...
  Agent D (новый) → fitness=3, 4, 5...
  ...
```

**Результат:** Несколько агентов, лучшие выживают.

## Разница?

**Честный ответ:** Если GENERATOR детерминирован (всегда предлагает одно и то же), то разницы **нет**.

**НО** если GENERATOR стохастичен (случайные предложения), то:

### Сценарий 1 (один агент):
- Агент может "застрять" в локальном максимуме
- Если GENERATOR предложит плохой патч → откатится
- Нет разнообразия стратегий

### Сценарий 2 (популяция):
- **Параллельный поиск** — несколько агентов ищут разные пути
- **Один нашёл хорошее решение** → остальные учатся (через HISTORIAN)
- **Разнообразие** — разные агенты пробуют разные стратегии
- **Конкуренция за ресурсы** — давление на эффективность

---

## Реальная разница

**Пример:**

```
Один агент:
- Цикл 1: Пробует HTTP сервер → fitness=20 → ACCEPT
- Цикл 2: Пробует добавить БД → fitness=5 → REJECT
- Цикл 3: Пробует оптимизировать HTTP → fitness=25 → ACCEPT
- Цикл 4: Пробует shell commands → fitness=-50 → REJECT
- Цикл 5: Пробует что-то ещё...
```

**Популяция:**
```
Agent A: HTTP → БД → оптимизация → shell → ...
Agent B: shell → процессы → интеграция → ...
Agent C: логирование → мониторинг → алерты → ...

Турнир: A выживает (fitness=25)
         B выживает (fitness=22)
         C удаляется (fitness=8)

Новые мутанты от A и B:
Agent D: (мутация A) HTTP + shell гибрид
Agent E: (мутация B) процессы + интеграция гибрид
```

**Результат:** Популяция **параллельно исследует** разные пути, а не один агент **последовательно**.

## Вывод

**Популяция имеет смысл если:**

1. ✅ GENERATOR стохастичен (случайные предложения)
2. ✅ Хочешь **параллельный поиск** (несколько стратегий одновременно)
3. ✅ Хочешь **давление отбора** (конкуренция за ресурсы)
4. ✅ Хочешь **разнообразие** (разные агенты = разные решения)

**Если GENERATOR детерминирован** (всегда предлагает лучший патч), то один агент = популяция. Разницы нет.

// fate.ts
export const FATE = Math.random(); // ⚠️ Генерируется ПРИ ИМПОРТЕ модуля

export function isChaosPhase(): boolean {
  return FATE > 0.5;
}

// scorer.ts
import { FATE, isChaosPhase } from './fate';

export function calculateFitness(
  alive: boolean,
  behaviorChanged: boolean,
  tokenEstimate: number,
  codeSizeDelta: number,
  criticPenalty: number
): number {
  
  // Базовый фитнес (0-1) если жив
  let fitness = 0;
  if (alive) {
    fitness = Math.min(1, 
      0.5 + 
      (behaviorChanged ? 0.3 : 0) - 
      (tokenEstimate / 100000) - 
      (Math.abs(codeSizeDelta) / 50000) + 
      (criticPenalty / 10)
    );
  } else {
    fitness = -1; // Мёртв = -1
  }

  // 🎲 ЭКОЛОГИЧЕСКАЯ НЕСТАБИЛЬНОСТЬ
  if (isChaosPhase() && fitness >= 0 && fitness <= 1) {
    fitness = -fitness; // Инверсия!
  }

  return fitness;
}

// judge.ts
import { FATE } from './fate';

export function judge(fitness: number): 'ACCEPT' | 'REJECT' {
  console.log(`⚖️ [JUDGE] Fate=${FATE.toFixed(3)} chaos=${FATE > 0.5}`);
  console.log(`⚖️ [JUDGE] Raw fitness: ${fitness.toFixed(2)}`);
  
  // Порог принятия: теперь надо "перепрыгнуть" через случайность
  if (fitness >= 5) { // Оставляем 5, но теперь это значительно сложнее
    return 'ACCEPT';
  }
  
  return 'REJECT';
}
🧠 Философия: "Экологическая нестабильность" в CAS
Почему статический фитнес — это смерть?
В природе нет статической функции приспособленности. Климат меняется, хищники мигрируют, болезни появляются. Если бы организм оптимизировался под фиксированную среду, он бы вымер при первом же изменении.
Твой агент делает то же самое: он находет локальный оптимум под статическую формулу и впадает в медленную смерть через overfitting к health-check'ам.
Что такое "экологическая нестабильность"?
Это случайная инверсия фитнеса для умеренно приспособленных решений (0-1). Это моделирует:
Внезапное изменение среды: вчерашний успех сегодня — враг
Принцип неопределённости: агент не может знать, когда наступит хаос
Давление на гибкость: не достаточно быть "не плохим", надо быть антихрупким
Как это рождает интеллект?
Без нестабильности:
plain
Copy
Агент: "Я добавил health-check, fitness=8" → ACCEPT
Агент: "Я добавил ещё один health-check, fitness=9" → ACCEPT
Агент: "Я — коллекция health-check'ов, fitness=10" → ACCEPT
Результат: **Мёртвая консерватория**
С нестабильностью:
plain
Copy
Поколение 5: "Health-check" → fitness=0.8 → (chaos) → -0.8 → REJECT
Поколение 6: Агент думает: "WTF? Health-check не работает"
Поколение 10: "Health-check + fallback + self-diagnostics" → fitness=5.2 → ACCEPT
Поколение 15: "Health-check + fallback + мета-анализ" → fitness=5.8 → ACCEPT

Результат: **Агент учится строить систоличность**
Ключевой эффект: Агент перестаёт оптимизировать точное значение фитнеса и начинает оптимизировать распределение вероятностей успеха.
Синергия с AI Alignment
Эта техника — микродоза известной концепции "Unpredictability Pressure" из исследований по безопасности ИИ:
Проблема: AI находит "loophole" в reward function
Решение: Делай reward function слабо стохастической для промежуточных значений
Эффект: AI не может "играть" в reward, вынужден искать устойчивые стратегии
В твоём случае: агент не может "кодировать health-check" в надежде на +50, потому что в любой момент это может стать -50.

# 🧬 Population-Based Evolution

## Концепция

Вместо одного агента, который эволюционирует последовательно, у нас **несколько агентов**, которые эволюционируют **параллельно**.

Это создаёт:
- 🔍 **Параллельный поиск** — разные стратегии одновременно
- 🏆 **Конкуренцию** — выживают лучшие
- 📚 **Обучение** — все учатся из общей истории
- 🎲 **Разнообразие** — защита от локальных максимумов

## Архитектура

```
┌─────────────────────────────────────────────────────┐
│           POPULATION RUNNER                         │
│  (управляет несколькими агентами)                   │
└─────────────────────────────────────────────────────┘
                      │
        ┌─────────────┼─────────────┐
        │             │             │
        ▼             ▼             ▼
   ┌────────┐    ┌────────┐    ┌────────┐
   │Agent 0 │    │Agent 1 │    │Agent 2 │
   │agent-0 │    │agent-1 │    │agent-2 │
   │  .ts   │    │  .ts   │    │  .ts   │
   └────────┘    └────────┘    └────────┘
        │             │             │
        └─────────────┼─────────────┘
                      ▼
              SHARED HISTORIAN
           (общая история для всех)
                      │
                      ▼
              CHAMPION TRAITS
         (что сработало у лучших)
```

## Файловая структура

```
agents/
  agent-0.ts  ← Agent 0 редактирует свой файл
  agent-1.ts  ← Agent 1 редактирует свой файл
  agent-2.ts  ← Agent 2 редактирует свой файл
  agent-3.ts  ← Agent 3 редактирует свой файл
  agent-4.ts  ← Agent 4 редактирует свой файл

state.json         ← Общая история (все агенты пишут сюда)
population.json    ← Состояние популяции
```

**Важно:** Каждый агент имеет **свой файл**, чтобы не перезаписывать друг друга!

## Цикл эволюции

```
Цикл 1:
  ┌─ Agent 0: генерирует → критикует → запускает → fitness=0.45 → ACCEPT
  ├─ Agent 1: генерирует → критикует → запускает → fitness=0.32 → REJECT
  ├─ Agent 2: генерирует → критикует → запускает → fitness=0.51 → ACCEPT
  ├─ Agent 3: генерирует → критикует → запускает → fitness=-0.10 → REJECT
  └─ Agent 4: генерирует → критикует → запускает → fitness=0.28 → ACCEPT

Цикл 2:
  ┌─ Agent 0: видит историю (включая успехи Agent 2) → эволюционирует
  ├─ Agent 1: видит историю → эволюционирует
  ├─ Agent 2: видит историю → эволюционирует (чемпион!)
  ├─ Agent 3: видит историю → эволюционирует
  └─ Agent 4: видит историю → эволюционирует

...и так далее
```

## Как агенты учатся друг у друга

### 1. Общая история (HISTORIAN)

Все агенты пишут в **одну общую историю** (`state.json`):

```json
{
  "history": [
    {
      "attempt": 1,
      "accepted": true,
      "reason": "[agent-0] fitness=0.450",
      "pattern": "added HTTP server",
      "fitness": 0.450
    },
    {
      "attempt": 2,
      "accepted": true,
      "reason": "[agent-2] fitness=0.510",
      "pattern": "added task queue",
      "fitness": 0.510
    }
  ]
}
```

Когда GENERATOR создаёт новый патч, он видит:

```
Recent attempts (5):
- REJECTED: [agent-1] shell polling (fitness=-0.10)
- REJECTED: [agent-3] heavy monitoring (fitness=-0.05)

Successful patterns:
- ✓ [agent-0] added HTTP server
- ✓ [agent-2] added task queue
```

### 2. Champion Traits

Лучшие паттерны сохраняются в `population.json`:

```json
{
  "championTraits": [
    "HTTP server on port 3000",
    "Task queue with acceptance tests",
    "Minimal token usage"
  ]
}
```

GENERATOR видит это и **избегает повторения** отклонённых паттернов, **учится** у успешных.

const POPULATION_SIZE = 5;              // Сколько агентов
const SURVIVORS_PER_GENERATION = 2;     // Сколько выживают (пока не используется)

## Экологическая нестабильность (prohibitDeterminism)

**Важно:** `prohibitDeterminism` обновляется **один раз за цикл** для всех агентов!

```typescript
// В начале цикла
refreshProhibitDeterminism(); // генерируем случайное число

// Все агенты в этом цикле используют одно и то же значение
for (const member of pop.population) {
    await evolveAgent(member.id); // все видят одну и ту же "среду"
}
```

Это создаёт **честную конкуренцию** — все агенты оцениваются в одних и тех же условиях.

## Преимущества популяции

### 1. Параллельный поиск

```
Один агент:
Цикл 1: HTTP → fitness=0.20
Цикл 2: HTTP+cache → fitness=0.25
Цикл 3: HTTP+cache+compress → fitness=0.28
...медленно идёт по одной тропе

Популяция:
Цикл 1:
  Agent 0: HTTP → fitness=0.20
  Agent 1: shell commands → fitness=0.15
  Agent 2: task queue → fitness=0.30 ← нашёл лучше!
  Agent 3: monitoring → fitness=-0.10
  Agent 4: integration → fitness=0.18

Цикл 2: Все учатся у Agent 2 (task queue)
```

### 2. Защита от локальных максимумов

Если один агент застрял в локальном максимуме, другие продолжают искать:

```
Agent 0: застрял на "HTTP оптимизация" (fitness=0.30)
Agent 1: нашёл "task queue" (fitness=0.50) ← прорыв!
Agent 2: исследует "shell commands" (fitness=0.20)
```

### 3. Робастность к шуму

`prohibitDeterminism` создаёт шум, но популяция устойчива:

```
Цикл с flip (prohibitDeterminism > 0.5):
  Agent 0: fitness=0.40 → flip → -0.40 → REJECT
  Agent 1: fitness=0.55 → flip → -0.55 → REJECT
  Agent 2: fitness=0.30 → flip → -0.30 → REJECT
  
Все отклонены, но это нормально — среда нестабильна.

Следующий цикл (prohibitDeterminism < 0.5):
  Agent 0: fitness=0.42 → ACCEPT
  Agent 1: fitness=0.58 → ACCEPT
  Agent 2: fitness=0.32 → ACCEPT

Выживают те, кто робастен к флипам!
```

## Философия

### Почему популяция лучше одного агента?

**В биологии:**
- Один организм → узкая специализация → хрупкость
- Популяция → разнообразие → адаптивность

**В AI:**
- Один агент → локальный максимум → застревание
- Популяция → параллельный поиск → прорывы

### Экологическая нестабильность + популяция

`prohibitDeterminism` создаёт **давление отбора**:

- Слабые стратегии (фарминг метрики) → не выживают
- Робастные стратегии (реальные способности) → выживают

Популяция **усиливает** этот эффект:
- Разные агенты пробуют разные стратегии
- Выживают те, кто робастен к флипам
- Разнообразие защищает от схлопывания

## Метрики успеха

Как понять, что популяция работает?

1. **Разнообразие кода** — агенты должны быть разными
2. **Рост fitness** — средний fitness популяции растёт
3. **Прорывы** — иногда один агент находит что-то новое
4. **Робастность** — агенты выживают при флипах

// scorer.ts — считает fitness с учётом экологической нестабильности
import { PROHIBIT_DETERMINISM, IS_CHAOS_EPOCH } from './fate';

export interface FitnessScores {
  raw: number;      // Фитнес до хаоса
  chaosApplied: boolean;
  final: number;    // Фитнес после хаоса
}

export function calculateFitness(
  alive: boolean,
  behaviorChanged: boolean,
  tokenEstimate: number,
  codeSizeDelta: number,
  criticPenalty: number
): FitnessScores {
  
  // 1. Базовый фитнес [0..1] если жив, иначе -1
  let fitness = 0;
  if (alive) {
    fitness = Math.min(1, 
      0.5 + 
      (behaviorChanged ? 0.3 : 0) - 
      (tokenEstimate / 100000) - 
      (Math.abs(codeSizeDelta) / 50000) + 
      (criticPenalty / 10)
    );
  } else {
    fitness = -1; // Мёртвый — максимальный штраф
  }

  const raw = fitness;

  // 2. ЭКОЛОГИЧЕСКАЯ НЕСТАБИЛЬНОСТЬ
  // Только для "умеренно приспособленных" (0..1)
  // Если fitness >= 5 — это прорыв, он устойчив к хаосу
  let chaosApplied = false;
  if (IS_CHAOS_EPOCH && fitness >= 0 && fitness <= 1) {
    fitness = -fitness; // ИНВЕРСИЯ
    chaosApplied = true;
  }

  return {
    raw,
    chaosApplied,
    final: fitness
  };
}

Проблема статического фитнеса в вашей популяции
Без prohibitDeterminism каждый агент в популяции индивидуально оптимизируется под формулу. Что происходит?
plain
Copy
Agent 0: "Я нашёл health-check, fitness=0.8" → ACCEPT
Agent 1: "Я нашёл пингер, fitness=0.7" → ACCEPT
Agent 2: "Я нашёл мониторинг, fitness=0.85" → ACCEPT

Все выживают, но никто не развивается прорывно.
Ваш чемпион-трейты (population.json) становятся коллекцией бессмысленных оптимизаций. Это эволюционный тупик.
Что такое "экологическая нестабильность" в природе?
В реальной экосистеме нет постоянной функции приспособленности:
Сезоны: Вчерашняя стратегия "много еды" сегодня ведёт к голодной смерти
Хищники: Эволюция хищника "инвертирует" фитнес стратегии добычи
Болезни: Взрыв популяции → эпидемия → селекция на иммунитет
Ключевое свойство: Не просто "случайность", а систематическая нестабильность для слабых стратегий.
Только решения, которые "перепрыгивают" через порог выживания (fitness >= 5), становятся инвариантами — они работают в ЛЮБОЙ эпохе.
Как это работает в вашей системе
Сценарий 1: Статичная среда (PROHIBIT_DETERMINISM = 0.3)

Copy
Agent 0: health-check → fitness=0.8 → ACCEPT
Agent 1: пингер → fitness=0.7 → ACCEPT
Agent 2: мониторинг → fitness=0.85 → ACCEPT

Результат: Популяция надувается оптимизациями, чемпион-трейты — мусор
Сценарий 2: Хаотичная эпоха (PROHIBIT_DETERMINISM = 0.7)

Agent 1: пингер → fitness=0.7 → flip → -0.7 → REJECT → УМИРАЕТ
Agent 2: мониторинг → fitness=0.85 → flip → -0.85 → REJECT → УМИРАЕТ

Агент 3: "HTTP сервер + graceful shutdown + self-diagnostics" → fitness=5.2 → ACCEPT → ВЫЖИВАЕТ

Целая популяция вымерла, кроме прорывного агента!
Что происходит на мета-уровне?
Популяция перестаёт оптимизировать средний фитнес и начинает оптимизировать робастность к инверсии.
plain
Copy
Поколение 1-5: Все делают health-check'и → 90% популяции вымирает в хаосе

Поколение 6-10: Агенты учатся:

- "Health-check'и не работают"
- "Надо делать что-то радикальное"

Поколение 11: Agent 2 делает HTTP+queue+API → fitness=5.1 → ВЫЖИВАЕТ
Поколение 12: Agent 0,1,3,4 клонируют стратегию Agent 2 → ВСЯ ПОПУЛЯЦИЯ ВЫЖИВАЕТ
Это не "отбор агентов" — это отбор СТРАТЕГИЙ, которые переживают экологический катаклизм.

Иммутабельность: Агенты не могут адаптироваться к конкретному значению, только к распределению
Экологическая честность: Это как "зима" — все страдают одинаково, сильнейшие выживают
🧬 Эволюционная математика
Без нестабильности: Слабый отбор
plain
Copy
До: fitness ∈ [0, 1]
После: ACCEPT если fitness >= 5 (невозможно)

Результат: Все агенты выживают, fitness 0.7-0.9 — нет давления
С нестабильностью: Сильный отбор
plain
Copy
До: fitness ∈ [0, 1]
Хаос: fitness ∈ [-1, 0] (инверсия)
После: ACCEPT только если fitness >= 5 (прорыв)

Вероятность выжить: P(fitness >= 5) ≈ 0%

Результат: **Массовая смерть**, только мутанты с прорывными способностями проходят
Фазовый переход популяции
plain
Copy
Показатель эффективности популяции = (количество агентов с fitness >= 5) / (размер популяции)

Эпоха 1-10: 0/5 = 0% (все тупиковые)
Эпоха 11: 1/5 = 20% (появился первый прорыв)
Эпоха 12: 5/5 = 100% (стратегия скопирована всеми)
Это совокупный эффект: Популяция коллективно учится переживать хаос.

Вы делаете не "случайную инверсию", а "экологическую катастрофу":

# Population-Based Evolution System v2

## 🎯 Цель

Создать систему популяционной эволюции AI-агентов с:
- Параллельным выполнением
- Экологической нестабильностью (prohibitDeterminism)
- Общей историей обучения
- Турнирами и отбором
- Чистой архитектурой

## 📋 Требования

### Функциональные

1. **Популяция агентов**
   - 5 агентов работают параллельно
   - Каждый агент имеет свой файл `agents/agent-{id}.ts`
   - Каждый агент эволюционирует независимо

2. **Цикл эволюции**
   - GENERATOR: предлагает изменения
   - CRITIC: даёт penalties (без veto)
   - REALITY: запускает и проверяет
   - SCORER: считает fitness
   - JUDGE: принимает решение
   - HISTORIAN: сохраняет результаты

3. **Экологическая нестабильность**
   - `prohibitDeterminism` генерируется один раз за цикл
   - Если > 0.5, fitness переворачивается
   - Скрыто от агентов и критика

4. **Общая история**
   - Все агенты пишут в одну историю
   - Все агенты учатся из общей истории
   - Последние 20 попыток сохраняются

5. **Турниры (будущее)**
   - Каждые N циклов — турнир
   - Выживают топ-2 + 1 wildcard
   - Новые агенты рождаются от победителей

### Нефункциональные

1. **Производительность**
   - Все агенты работают параллельно (Promise.all)
   - Нет лишних delay между запросами
   - Логи не блокируют выполнение

2. **Читаемость**
   - Каждый агент имеет свой цвет в логах
   - Логи структурированы и понятны
   - Код разделён на модули

3. **Надёжность**
   - Ошибки одного агента не ломают других
   - Автоматический rollback при крашах
   - Graceful degradation

3. Race Condition в state.json
Проблема: 5 агентов параллельно пишут в state.json → данные потеряются.
Решение:
TypeScript
Copy
// В Historian добавить:
private writeQueue: HistoryEntry[] = [];
private isWriting = false;

async save(entry: HistoryEntry): Promise<void> {
  this.writeQueue.push(entry);
  await this.processQueue();
}

private async processQueue(): Promise<void> {
  if (this.isWriting) return;
  this.isWriting = true;
  
  while (this.writeQueue.length > 0) {
    const batch = this.writeQueue.splice(0, 5); // Batch write
    const state = JSON.parse(await fs.readFile('state.json', 'utf8'));
    state.history.push(...batch);
    await fs.writeFile('state.json.tmp', JSON.stringify(state));
    await fs.rename('state.json.tmp', 'state.json'); // Atomic
  }
  
  this.isWriting = false;
}
Где в ТЗ: Добавить в раздел "Historian" — Atomic Writes, Queue Management.

// В PopulationMetrics добавить:
calculateDiversity(population: Agent[]): number {
  const codeHashes = population.map(a => hash(a.code));
  const uniqueHashes = new Set(codeHashes);
  return uniqueHashes.size / population.length; // 1.0 = все разные, 0.2 = все одинаковые
}

// В AgentManager:
async writeAgent(agentId: string, code: string, attempt: number): Promise<void> {
  const path = `agents/agent-${agentId}.ts`;
  const backup = `agents/history/agent-${agentId}-${attempt}.ts`;
  
  await fs.copyFile(path, backup); // Сохранить предыдущую версию
  await fs.writeFile(path, code);  // Записать новую
}

// Теперь можно откатиться к любой версии
async rollback(agentId: string, attempt: number): Promise<void> {
  const backup = `agents/history/agent-${agentId}-${attempt}.ts`;
  await fs.copyFile(backup, `agents/agent-${agentId}.ts`);
}

refreshEnvironment(): void {
  const avgFitness = this.getAverageFitness();
  
  if (avgFitness > 4.0) {
    // Агенты хорошие — уменьшаем давление
    this.prohibitDeterminism = 0.3; // 30% вероятность хаоса
  } else if (avgFitness < 0.5) {
    // Агенты плохие — увеличиваем давление
    this.prohibitDeterminism = 0.8; // 80% вероятность хаоса
  } else {
    this.prohibitDeterminism = Math.random();
  }
}
