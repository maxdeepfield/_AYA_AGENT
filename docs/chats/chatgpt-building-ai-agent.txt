You can’t write “an agent that can do ANYTHING.” You can write an agent that can attempt anything within explicit permissions, by routing tasks to tools it’s allowed to use. The difference is not marketing; it’s reliability and liability.

What “do anything” really means

An agent is four parts:

Intent router: turns a messy request into a concrete goal + constraints.

Planner: breaks goal into steps, chooses tools, sets success criteria.

Executor: runs tools, handles errors, retries, and confirms outcomes.

Governor: permissions, safety rules, budgets, logging, and human approval.

If you skip (4), you don’t get “do anything,” you get “randomly destroy stuff.”

Minimal architecture that actually works
1) Define a capability boundary (non-negotiable)

Pick what “anything” means for your environment:

Local computer actions (files, apps)

Web actions (browser automation)

API actions (GitHub, email, calendar, cloud)

Coding (repo changes, tests, deployments)

Content work (docs, reports)

Each capability must be behind a tool interface with:

Inputs validated (schema)

Output captured (structured)

Side effects declared (what it changes)

2) Use a tool-first loop (not pure chat)

A reliable agent runs an explicit loop:

Parse task → produce structured objective

Plan steps → each step is “call tool X with args Y”

Execute → capture outputs

Verify → tests, diffs, screenshots, status checks

Summarize → what changed, where, how to rollback

3) Add a permissions model

Implement “least privilege”:

Read-only by default

Write requires approval (or scoped tokens)

Destructive actions require explicit confirmation (delete, money, sending emails, prod deploy)

Do not put admin keys directly in the agent. Use short-lived tokens or a broker service.

4) Give it memory, but only the right kind

Two memories:

Working memory (per task): plan, intermediate results.

Long-term memory (opt-in): stable prefs, project facts, tool configs.

Also keep an event log of actions for auditing and rollback.

5) Make it verifiable

If it can’t prove it succeeded, it didn’t.
Verification examples:

Code: tests + lint + build artifacts

Web: URL + screenshot + extracted key text

Files: checksum, diff, file tree

APIs: status codes + follow-up GET

The “tool belt” you actually need

Start with a small set; expand only after it’s stable.

Core tools

Filesystem: read/write/list, diff

Shell runner: run commands with sandboxing

Git: status/diff/commit/PR

HTTP client: call APIs

Browser automation (optional): Playwright-style actions

Safety tools

Secret manager

Policy checker (pre-flight)

Rate/budget limiter

Approval gate (UI or CLI prompt)

Hard truth: the main failure mode

Agents fail because:

They don’t have grounded state (they hallucinate files/pages/results)

Tools return errors and the agent “pretends” it worked

No verification step

Too-broad permissions early

Your first milestone is not “do anything.” It’s:
“Do 3 things end-to-end, perfectly, with logs and rollback.”

Build plan (fastest path)

Implement the loop: objective → plan → execute → verify → report.

Add 5 tools max.

Add approvals for anything that writes/deletes/sends.

Add evaluation: a folder of test tasks the agent must complete repeatedly.

“Total freedom” is not an implementation detail. It is a governance problem.

If you give an agent total freedom, you are the safety layer—whether you acknowledge it or not. Here is the reality, without softening it.

Why “total freedom” is a trap

Freedom without structure collapses into one of three outcomes:

Harmless impotence
The agent talks, plans, speculates—does nothing real.

Silent self-corruption
It learns local shortcuts, breaks invariants, accumulates hidden damage.

Catastrophic action
Deletes data, leaks secrets, spends money, locks you out.

There is no fourth outcome. Anyone claiming otherwise is selling hype.

The correct reframing

You do not give freedom.

You give:

Maximum action surface

Minimum irreversible authority

Explicit escalation paths

Freedom is emergent, not granted.

What “maximum freedom” actually looks like (in practice)
1. Everything is possible, nothing is implicit

The agent can:

Call any API you expose

Receive any webhook

Chain tools arbitrarily

Propose new goals

But it cannot:

Create new permissions

Invent credentials

Skip verification

Bypass logging

This is how real autonomous systems work (finance, infra, aviation).

2. Authority is tiered, not binary

You need levels, not “allowed / not allowed”.

Example:

Level 0: Observe only

Level 1: Write drafts, comments, branches

Level 2: Merge, deploy to staging

Level 3: Touch production / money / deletions

Level 4: Meta-actions (changing its own tools or policies)

Promotion between levels requires:

Evidence

Time

Or explicit human approval

This is not paranoia. This is how you keep control while scaling autonomy.

3. Self-modification is the real danger

“Total freedom” usually means:

“Let it rewrite itself.”

This is where systems die.

If you allow self-modification:

Separate core from extensions

Core cannot be edited by the agent

Extensions are versioned, reviewable, revertible

No exceptions.

4. The agent must be legible

A free agent that cannot explain itself is already unsafe.

Every action must produce:

Why it acted

What it changed

What it expects next

How to undo it

If it can’t do this, it doesn’t act.

The paradox you must accept

Total freedom requires maximum constraint.

Constraints are not limitations.
They are what make autonomy stable over time.

Every real autonomous system—biological, technical, organizational—works this way.

Design a kill-switch architecture
Kill-switch architecture for a tool-using agent

Goal: at any moment, you can force the agent into a safe state, stop ongoing actions, and prevent new actions, even if the agent is buggy or hostile.

Design principles

Out-of-band control: kill-switch must not depend on the agent’s own process or credentials.

Fail-closed: on uncertainty, expired leases, missing heartbeats → agent cannot act.

Centralized enforcement: all side effects must pass through a single enforcement point.

Fast + durable: immediate stop + durable lock that survives restarts.

Auditable: every kill/un-kill is logged with who/when/why.

Core components
1) Tool Gateway (the choke point)

All external actions go through a gateway you control:

HTTP proxy for outbound API calls

Command runner service for shell actions

Git/CI dispatcher

Browser automation runner

The agent never directly talks to Stripe/GitHub/cloud/etc. It only talks to Tool Gateway.

Gateway enforces:

Kill state

Rate limits

Allowed tool list

Argument validation

Token brokering (see below)

2) Control Plane (authoritative state)

A tiny service (or even a highly-available KV) that stores:

kill_state = {ON|OFF}

per-agent states: disabled, read_only, write_ok, etc.

reason, actor, timestamp

escalation state (e.g., “maintenance mode”)

This control plane must be reachable by Tool Gateway even if the agent is down.

3) Lease-based authorization (deadman’s switch)

The agent must hold a short-lived lease to execute side effects.

Lease TTL: 30–120 seconds

Renewed by heartbeat

If heartbeat stops → lease expires → gateway denies all write actions

This prevents “zombie agents” and stuck loops from continuing.

4) Token Broker (no long-lived secrets in the agent)

The agent never stores real API keys.

Tool Gateway requests short-lived tokens from broker

Broker issues scoped tokens only if kill_state=OFF and lease valid

Tokens are:

short-lived

tool-scoped

action-scoped when possible

If you flip kill_state=ON, broker stops issuing tokens immediately.

5) Two-layer kill switch

Implement both:

Soft kill (logical):

Control plane sets kill_state=ON

Gateway denies side effects, allows read-only

Hard kill (physical/process):

Stops compute executing agent + tool runners

Options:

stop containers/VMs (or scale to zero)

revoke network egress at firewall level

revoke service account permissions at IAM

disable broker credentials

Soft kill is instant and safe. Hard kill is the “nuke it” option.

Enforcement model
Action categories

Every tool is tagged:

Read: safe queries, logs, status, fetch PRs

Write: create branch, comment, update config

Destructive: delete resources, revoke creds, merge main, prod deploy, money movement

Kill behavior:

Kill ON → allow Read only

Write/Destructive denied

Optional: allow “cleanup-only” actions if signed and explicitly permitted (see below)

Cleanup exception (optional but useful)

Sometimes you need the agent to stop safely (cancel jobs, release locks).
Implement a Cleanup Mode:

A separate tool category: “cleanup”

Only callable by Tool Gateway itself, not the agent

Triggered automatically when kill_state flips ON:

cancel CI jobs

stop deployments

unlock resources

terminate browser sessions

This prevents “kill = leave mess”.

Kill-switch interfaces (multiple, redundant)
A) CLI / Admin UI

agentctl kill on --reason "suspected loop"

agentctl kill off --reason "resolved"

Protected by strong auth (SSO, hardware key)

B) Break-glass manual switch

A separate mechanism that does not rely on your main auth stack:

A pre-generated one-time recovery code stored offline

Or a physically separate admin account

Used when SSO/IAM is compromised.

C) Network-level egress cut

A firewall rule / security group / proxy denylist that can block:

all outbound internet

all outbound except the control plane

or only “danger endpoints” (Stripe, cloud delete APIs)

This is your last resort when software control is compromised.

Safety invariants (must be true)

No direct credentials in agent

No direct internet from agent (agent talks only to gateway)

Gateway checks kill state on every write/destructive call

Write requires valid lease

Lease renewal requires independent heartbeat

All actions logged at gateway with request/response metadata

Kill state stored durably and replicated

Hard kill available outside the agent’s environment

If any invariant fails, you don’t have a kill switch; you have wishful thinking.

Failure modes and how this design handles them
Agent goes rogue / loops

Lease expires if heartbeat fails

Even with heartbeat, you flip kill_state=ON → gateway denies side effects

Agent compromised

Attacker can’t use long-lived API keys (none exist)

Token broker refuses new tokens when killed

Network egress cut stops everything

Tool Gateway compromised

Hard kill via network/IAM revocation still works

Run gateway with least privilege; isolate it

Monitor for policy changes

Control plane unavailable

Fail-closed:

If gateway cannot read control plane state → deny writes

Recommended default settings

Lease TTL: 60 seconds

Heartbeat period: 10 seconds

Kill check: on every write/destructive tool call

Read calls: allowed even during kill

Cleanup mode: enabled, but only gateway-driven

Logging: immutable append-only (or write-once storage)

Minimal “v1” implementation scope (fastest that’s real)

Tool Gateway: accepts tool requests, blocks/permits by kill_state

Control Plane: a durable key-value store with kill_state

Lease: in control plane with TTL

No token broker initially: still possible if you keep secrets only in gateway (not agent)

Hard kill: container/VM stop + firewall egress rule

That already gives you a functional kill switch.
