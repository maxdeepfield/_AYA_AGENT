## Лучший вариант реализации, который у нас “выкристаллизовался” 🧬

Если собрать **всё, что реально работало** в твоих текущих прототипах (само-починка, rollback, компиляция/смоук-тесты) и **всё, что ломало прогресс** (критик-veto, “мониторинг ради мониторинга”, capability-cosplay типа ML/RL/etherscan, раздувание кода, уход в мета-парсеры), то самый сильный итоговый дизайн такой:

**Не один “самопереписывающийся терминатор”, а популяция агентов + измеримые способности + отбор по фитнесу + шум среды.**
Это даёт рост “в сторону функций”, а не “в сторону самокомфорта”.

---

## 1) Ключевая идея: Body vs Brain

### 🧱 Body (Population Runner) — “мир и законы”

Это твой код, которому ты доверяешь:

* запускает агентов (worker threads),
* ограничивает ресурсы (время/токены/сеть/процессы),
* наблюдает поведение,
* считает fitness,
* решает судьбу (accept/reject, турниры, размножение),
* один пишет историю (single-writer).

### 🧠 Brain (Agent) — “мутирует”

Агент **не должен** сам себе выдавать полномочия. Он:

* предлагает изменения в своём файле,
* эмитит события,
* пытается заработать очки фитнеса **через новые реальные способности**, а не через болтовню.

---

## 2) Почему именно так (из твоих примеров)

1. **Самоулучшение одного агента** быстро начинает оптимизировать “выживание без падений”: парсеры ошибок, метрики, blacklists ✅
   Но затем часто уходит в **рефактор/раздувание**, потому что это “безопасные” улучшения.

2. Когда ты дал интернет/интеграции — модель начала лепить “фичи-фантазии” (ML/RL/blockchain) ❌
   То есть без внешнего “мира” и проверок агент начинает **симулировать** способности.

3. **Критик-veto** у тебя фактически стал “богом”, который отвергает любые микро-шаги как “cosplay” ❌
   Вывод: критик должен быть **penalty**, а не “убийца прогресса”.

Итог: нужен **формальный мир** (runner), где “реальная фича” = **сигнал + тест**, а не объяснение в промпте.

---

## 3) Исполняемый контракт: события + acceptance tests ✅

### Event Protocol (минимум)

Агент обязан общаться структурированно (иначе он не может доказать, что сделал что-то полезное):

* `agent:alive`, `agent:ready`
* `capability:registered { name, version?, details? }`
* `task:*` (created/started/completed/failed)
* опционально: `server:listening`, `proc:spawned`, `http:external`

### Acceptance Tests

Каждая “способность” засчитывается **только если runner смог проверить**.

Примеры:

* `http_server`: агент заявил capability + runner сделал GET `/status` = 200
* `task_queue`: runner отправил команду “создай задачу” → агент завершил `task:completed`
* `process_manager`: агент заспавнил allowlisted команду → runner увидел `proc:exit` корректный

Это убивает capability-cosplay лучше любого критика.

---

## 4) Fitness: не “нравится/не нравится”, а экономика 🪙

Фитнес должен награждать **новые проверенные способности** и штрафовать **цену**.

**Что считаем валютой:**

* токены (приближенно, хотя бы `chars/4`)
* размер кода (delta)
* внешние вызовы (HTTP, spawn)
* время выполнения
* количество событий (как прокси активности)

**Критик** возвращает только JSON-penalty (0, -1, -2, -3) и причины.
Никаких veto.

---

## 5) Ecological Instability: `prohibitDeterminism` 🌪️

* Один раз на поколение генерируется скрытая переменная `prohibitDeterminism ∈ [0,1)`.
* Если `> 0.5` — итоговый fitness флипается знаком.

Зачем это реально нужно (без философии ради философии):

* ломает “тонкую подгонку” под текущую метрику,
* заставляет искать стратегии, которые устойчивы к шуму,
* в популяции это работает лучше, чем у одного агента: кто-то всё равно найдёт ход.

Важно:
`prohibitDeterminism` **не логируется публично** и **не попадает в prompts**.

---

## 6) Популяция: параллельный поиск + отбор

### Почему популяция лучше “прогнать 1 цикл 5 раз”

Потому что:

* популяция сохраняет **несколько линий** (diversity),
* сравнение идёт в одной среде (одно `prohibitDeterminism`),
* победивший “трейд” распространяется через historian.

### Базовая схема (лучший практический минимум)

* POP = 5 агентов
* В каждом поколении все получают шанс мутировать
* Затем **турнир** раз в N циклов:

  * выживают top-2 + 1 wildcard
  * остальные перерождаются (копия чемпиона/вице-чемпиона + мутация)

Wildcard нужен, чтобы не схлопнуться в клоны.

## 8) Режимы мира (чтобы можно было “дать рут”, но управляемо) 😈

Добавляем переключатель режима:

* **SAFE**: no external HTTP, spawn allowlist, строгие лимиты
* **WILD**: разрешены некоторые интеграции (OSV, локальные сервисы), но с бюджетами
* **CHAOS**: почти всё можно, но hard caps остаются (time/mem/disk/process count)

Так ты получаешь “весело”, но не получаешь “сам себе бан во всём интернете”.

* **JUDGE** вообще лучше держать детерминированным (правила + пороги), чтобы LLM не стала “политиком”.

// src/population/judge.ts
export function judgeSelection(
  candidates: Agent[],
  rules: TournamentRules
): Agent[] {
  // Pure math. No LLM. No interpretation.
  const sorted = candidates
    .map(a => ({
      ...a,
      score: (a.fitness.survival * a.fitness.novelty) / (a.fitness.efficiency + 1),
    }))
    .sort((a, b) => b.score - a.score);
  
  // Top 2 + wildcard (lowest efficiency = room to improve)
  const top = sorted.slice(0, 2);
  const wildcard = sorted.sort((a, b) => a.fitness.efficiency - b.fitness.efficiency)[0];
  
  return [...top, wildcard];
}

// src/population/tournament.ts
export function selectSurvivors(agents: Agent[]): Agent[] {
  // Сортируем по fitness (не по возрасту, не по случайности)
  const ranked = agents.sort((a, b) => b.fitness - a.fitness);
  
  const champions = ranked.slice(0, 2); // Топ-2 точно выживают
  const randomVictim = ranked[Math.floor(Math.random() * ranked.length)]; // Случайная жертва
  
  // Все остальные умирают и перерождаются как мутантные клоны чемпионов
  const dead = agents.filter(a => 
    !champions.includes(a) && a !== randomVictim
  );
  
  dead.forEach(agent => {
    const parent = champions[Math.floor(Math.random() * champions.length)];
    agent.code = mutate(parent.code); // Копируем чемпиона + мутации
    agent.fitness = 0; // Сбрасываем накопленное
  });
  
  return [...champions, ...dead];
}

export async function calculateFitness(agent: Agent): Promise<number> {
  let score = 0;
  
  // 1. Выживание (бинарно, но недостаточно)
  score += agent.alive ? 10 : -1000;
  
  // 2. ВАЛИДАЦИЯ (главное)

    const validation = await validate(cap);
    score += validation; // -30 до +50
  }
  
  // 3. Эффективность (чем меньше ресурсов, тем лучше)
  const efficiency = 1 / (agent.tokenCost + agent.memoryUsage + 1);
  score += efficiency * 10;
  
  // 4. Новизна (штраф за копирование)
  const similarityToOthers = calculateCodeSimilarity(agent, otherAgents);
  score -= similarityToOthers * 20; // Если код похож на соседа - штраф
  
  // 5. Турнирная нормализация (важно!)
  // Fitness становится относительным: лучший агент эпохи = 1.0, худший = 0.0
  return normalize(score, minScore, maxScore);
