# Runtime Loop

Этот документ описывает самый простой runtime loop для первого прототипа.
Не полный orchestration engine.
Не большая агентная система.

Только минимальный цикл, который должен дать существу:

- одно текущее дело
- один следующий шаг
- память о последствиях
- смену поведения после пустых повторов

## Цель Runtime Loop

Новый loop должен делать ровно одно:

превращать текущие предпочтения и текущее дело в один bounded step, а потом менять следующий выбор под реальный результат этого шага.

Если loop не делает этого, он не нужен.

## Общая Форма

Каждый тик выглядит так:

1. Проверить текущее состояние.
2. Уточнить или создать `currentObjective`.
3. Сгенерировать несколько возможных маленьких шагов.
4. Оценить шаги через drives, novelty и repeat penalty.
5. Выполнить один шаг.
6. Записать последствие.
7. Обновить objective.
8. Либо продолжать, либо остановиться.

## Состояния Loop

Для первого прототипа достаточно четырёх runtime-состояний:

```ts
type RuntimePhase =
  | 'idle'
  | 'select_step'
  | 'execute_step'
  | 'update_after_step';
```

### Смысл

- `idle`: нет немедленного шага, но система готова к следующему выбору
- `select_step`: выбирается следующий bounded step
- `execute_step`: шаг выполняется
- `update_after_step`: результат шага меняет состояние

Это не большие философские фазы.
Это просто технический loop.

## Тик

```ts
function tick(state: AgentSnapshot): AgentSnapshot {
  state = ensureObjective(state);
  const candidates = proposeCandidates(state);
  const selected = selectCandidate(state, candidates);

  if (!selected) {
    return handleNoStep(state);
  }

  const step = executeStep(state, selected);
  return applyStepConsequences(state, step);
}
```

## 1. ensureObjective

На каждом тике система должна знать, что сейчас является её текущим делом.

### Правила

- если `currentObjective` есть и он `active`, оставить его
- если objective `saturated`, завершить или заменить
- если objective `blocked`, создать новый только если blocker не требует user input
- если objective нет, создать минимальный objective из:
  - последнего пользовательского направления
  - или strongest drive
  - или ближайшего доступного объекта среды

### Важно

Новый objective не должен возникать из пустой абстракции вроде “мне тревожно”.
Он должен возникать из:

- объекта
- направления
- или конфликта drives

## 2. proposeCandidates

Система должна порождать не длинные планы, а несколько маленьких реальных шагов.

### Ограничение

На один тик достаточно `2..5` кандидатов.

### Примеры кандидатов

- прочитать один файл
- сделать один web search
- сравнить два последних результата
- выписать один список критериев
- задать один точный вопрос
- подождать, если сейчас нет хорошего шага

### Важно

Кандидаты должны быть bounded.
Если шаг нельзя завершить за один короткий акт, он слишком большой.

## 3. selectCandidate

Выбор шага не должен быть магическим.
Достаточно простой scoring-модели.

```ts
score =
  driveSupport
  + estimatedNovelty
  - estimatedRisk
  - repeatPenalty
```

### Что сюда входит

- `driveSupport`: насколько шаг поддерживает текущие drives
- `estimatedNovelty`: насколько он обещает реальное новое изменение
- `estimatedRisk`: риск бесполезного или разрушительного шага
- `repeatPenalty`: штраф за похожие уже сделанные шаги

### Критическое правило

Шаг, который уже несколько раз был пустым, должен почти перестать выбираться.

Именно это защищает систему от ритуального loop.

## 4. executeStep

После выбора делается один шаг.

### Разрешённые типы

- `read`
- `terminal`
- `web`
- `write`
- `respond`
- `wait`

### Принцип

Один тик = один шаг.

Не два.
Не цепочка из семи штук.
Не скрытая мини-эпопея.

Если нужен второй шаг, он должен быть результатом следующего тика.

## 5. applyStepConsequences

Это самый важный узел loop.

Именно здесь система решает, изменилось ли что-то реально.

### После шага нужно обновить

- `lastStep`
- `recentConsequences`
- `RepeatTracker`
- `currentObjective.progress`
- веса drives, если шаг реально что-то изменил

### Outcome-классы

Для старта достаточно таких outcome:

- `useful`
- `empty`
- `failed`
- `blocked`

### Правила

#### useful

- progress слегка растёт
- repeat penalty не растёт
- objective остаётся активным или доходит до done

#### empty

- progress не растёт
- repeat penalty растёт
- если таких шагов накопилось несколько, objective становится `saturated`

#### failed

- progress не растёт
- repeat penalty растёт
- риск похожих шагов повышается

#### blocked

- objective может перейти в `blocked`
- если blocker специфичен, разрешается один точный вопрос пользователю

## 6. handleNoStep

Иногда система не должна форсировать шаг.

Если хорошего шага нет, это лучше, чем ещё один пустой search.

### Минимальное поведение

Если ни один кандидат не проходит порог:

- система уходит в короткий `wait`
- objective остаётся
- на следующем тике выбор делается снова

### Важно

`wait` не должен интерпретироваться как трагедия.
Это просто отсутствие хорошего bounded step прямо сейчас.

## 7. Когда Спрашивать Пользователя

Пользователь не должен быть частью каждого тика.

Система спрашивает только если:

- objective реально заблокирован без внешнего входа
- есть узкая, конкретная недостающая информация
- без неё следующий шаг не выбрать честно

### Нельзя

- спрашивать просто потому, что система неуверенна
- спрашивать ритуально
- спрашивать “о чём поговорим?”
- спрашивать “какой tool использовать?”

### Можно

- один короткий вопрос по blocker
- один выбор из 2-3 реально различающихся веток

## 8. Saturation

Без saturation loop быстро умирает в повторе.

### Простое правило

Objective становится `saturated`, если:

- несколько похожих шагов подряд дали `empty`
- новизна не растёт
- candidate pool сжимается до почти одинаковых шагов

### Что делать дальше

После saturation система должна не повторять ещё один сбор, а выбрать одно из трёх:

- `synthesize`
- `reframe objective`
- `wait`

Это и есть первая настоящая защита от ритуала.

## 9. Minimum Synthesis

Даже в первом прототипе нужен очень маленький синтез.

Не большой reflection block.
Просто короткий артефакт вроде:

- что уже найдено
- что повторяется
- что теперь считается следующим лучшим направлением

### Пример

```ts
interface SynthesisNote {
  objectiveId: string;
  facts: string[];
  repeatsDetected: number;
  nextBestShift: string;
}
```

Этого уже достаточно, чтобы loop менял форму, а не только длился.

## 10. Самый Простой Runtime Contract

Если сжать весь loop до одной формулы:

- держать одно текущее дело
- делать один bounded step
- записывать последствие
- делать повтор дорогим
- менять objective только по реальному основанию

## Чего Тут Пока Нет

В этом loop сознательно нет:

- психологических состояний как центра системы
- больших reflection-циклов
- многослойного planner'а
- длинных монологов о себе
- огромной памяти

Если этот минимальный loop не даст нам связной непредсказуемости, усложнение почти наверняка только замаскирует проблему.

## Главный Тезис

Runtime loop должен не заставлять существо выглядеть живым.
Он должен делать так, чтобы следующий шаг реально зависел от последствий предыдущего.

Именно с этого начинается что-то похожее на хотение, а не на ритуал.
