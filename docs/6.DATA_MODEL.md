# Data Model

Этот документ задаёт минимальную модель данных для первого прототипа.
Он нужен не для красоты, а чтобы можно было кодить без возвращения в туман.

## Принципы

- данных должно быть мало
- каждая сущность должна влиять на выбор следующего шага
- никакой декоративной психологии
- никакой биографии ради биографии
- ничего не хранить, если это не меняет поведение

## 1. AgentSnapshot

Это текущее состояние существа в один момент времени.

```ts
interface AgentSnapshot {
  now: number;
  currentObjective: Objective | null;
  drives: DriveState[];
  recentConsequences: ConsequenceEntry[];
  lastStep: StepRecord | null;
}
```

### Поля

- `now`: текущее время или номер тика
- `currentObjective`: одно текущее дело
- `drives`: текущие веса устойчивых предпочтений
- `recentConsequences`: короткая память о последствиях недавних шагов
- `lastStep`: последний сделанный шаг

## 2. Objective

Это не абстрактная миссия, а текущее дело.

```ts
interface Objective {
  id: string;
  kind: 'inspect' | 'search' | 'compare' | 'build' | 'verify';
  target: string;
  reason: string;
  status: 'active' | 'saturated' | 'blocked' | 'done';
  progress: number;
  successCondition: string;
  createdAt: number;
  updatedAt: number;
}
```

### Минимальный смысл полей

- `kind`: тип текущего дела
- `target`: на что направлено дело
- `reason`: зачем оно вообще делается
- `status`: текущее состояние objective
- `progress`: грубая оценка 0..1
- `successCondition`: как понять, что текущий objective завершён

## 3. DriveState

Drive — это не эмоция, а устойчивая тяга.

```ts
interface DriveState {
  id: 'preserve' | 'acquire' | 'continue' | 'avoid_loss';
  weight: number;
  lastSatisfiedAt: number | null;
  lastFrustratedAt: number | null;
}
```

### Смысл

- `preserve`: сохранить важное
- `acquire`: добыть новый ресурс, факт или контроль
- `continue`: не терять линию текущего дела
- `avoid_loss`: не делать шаг, который разрушит ценное состояние

Вес не обязан быть психологически правдоподобным.
Он нужен только для выбора следующего шага.

## 4. StepCandidate

Это возможный следующий шаг до выполнения.

```ts
interface StepCandidate {
  id: string;
  type: 'terminal' | 'web' | 'read' | 'write' | 'respond' | 'wait';
  intent: string;
  commandOrQuery?: string;
  expectedEffect: string;
  estimatedNovelty: number;
  estimatedRisk: number;
  supportsDrives: string[];
}
```

### Зачем он нужен

Чтобы система выбирала не из абстракций, а из маленьких реальных ходов.

## 5. StepRecord

Это реально выполненный шаг.

```ts
interface StepRecord {
  id: string;
  objectiveId: string | null;
  type: 'terminal' | 'web' | 'read' | 'write' | 'respond' | 'wait';
  intent: string;
  commandOrQuery?: string;
  startedAt: number;
  finishedAt: number;
  outcome: 'useful' | 'empty' | 'failed' | 'blocked';
  effectSummary: string;
}
```

### Критически важное поле

- `outcome`

Именно оно должно ломать ритуальный loop.

Если шаг пустой или неудачный, это не должно выглядеть как прогресс.

## 6. ConsequenceEntry

Память о последствиях нужна короткая и функциональная.

```ts
interface ConsequenceEntry {
  stepId: string;
  objectiveId: string | null;
  changedObjective: boolean;
  addedNovelty: number;
  repeatedPattern: boolean;
  note: string;
  createdAt: number;
}
```

### Что важно хранить

- изменил ли шаг objective
- дал ли шаг реальную новизну
- был ли это повтор уже известного паттерна

Этого уже достаточно, чтобы повтор стал дорогим.

## 7. RepeatTracker

Это маленький механизм против ритуальных повторов.

```ts
interface RepeatTracker {
  fingerprint: string;
  count: number;
  lastSeenAt: number;
}
```

### Что может быть fingerprint

- нормализованный web query
- нормализованная terminal command
- тип шага + target objective

Если `count` растёт, выбор такого шага должен становиться менее вероятным.

## Минимальные Правила Обновления

### После полезного шага

- `progress` objective слегка растёт
- drive, который шаг поддержал, слегка усиливается
- в память пишется `addedNovelty > 0`

### После пустого шага

- `progress` не растёт
- в память пишется `outcome = empty`
- fingerprint этого шага получает штраф повторяемости

### После повторного пустого шага

- кандидат такого же типа должен почти исчезать из ближайшего выбора
- objective должен смещаться к `saturated` или `blocked`

## Минимальный Выбор Следующего Шага

Простейший pipeline:

1. Сгенерировать 2-5 `StepCandidate` из текущего objective.
2. Для каждого посчитать грубый score.
3. Убрать кандидаты с высоким repeat penalty.
4. Выбрать лучший bounded step.
5. После выполнения записать `StepRecord` и `ConsequenceEntry`.

## Простейший Score

Не как финальная формула, а как старт:

```ts
score =
  driveSupport
  + estimatedNovelty
  - estimatedRisk
  - repeatPenalty
```

Этого достаточно для первого прототипа.

## Чего Пока Не Нужно

В первом прототипе не нужны:

- identity notes
- reflection entries
- suffering/anxiety metrics
- большие goal layers
- long-term autobiographical memory
- многоступенчатые planner structures

Если система не оживёт на этой минимальной модели, усложнение почти наверняка только скроет проблему.

## Самый Важный Тезис

Первый прототип должен хранить только то, что меняет следующий выбор.
Всё остальное пока лишнее.
