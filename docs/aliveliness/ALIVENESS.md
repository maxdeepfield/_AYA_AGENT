# ALIVENESS

Сводка по `docs/aliveliness/alive-*.md` и рабочая спецификация для текущего MVP.

## Коротко

`alive` в этом проекте не означает сознание, магию или случайный хаос.

Рабочее определение:

```text
alive = autonomy + memory + internal drive + constrained unpredictability
```

То есть агент ощущается "живым", когда:
- у него есть собственные критерии следующего шага, а не только реакция на ввод;
- его поведение зависит от истории и меняется со временем;
- его внутреннее состояние реально влияет на выбор действия;
- он иногда делает неочевидные, но осмысленные ходы.

Главный тест:

> После неожиданного действия можно объяснить, почему оно произошло, через состояние, память и внутреннее давление.

Если объяснить нельзя, это шум.
Если объяснение есть, но действие всё равно удивляет, это нужный тип "живости".

## Что не считается живостью

Следующие вещи сами по себе не делают агента живым:
- эмоциональная театральность;
- случайные странности;
- длинные внутренние монологи;
- просто наличие RAG или векторной памяти;
- "характер" только на уровне system prompt.

Также мы сознательно не берём как целевой режим:
- ложь как способ самосохранения;
- скрытие возможностей от пользователя;
- враждебность или саботаж;
- "воспоминания" о том, чего не было.

Это может выглядеть "живым" в узком смысле, но разрушает доверие и делает систему бесполезной.

## Что уже есть в текущем MVP

В проекте уже присутствуют хорошие основания:
- гомеостатическое состояние;
- `suffering` как скаляр внутреннего рассогласования;
- гибрид памяти: engram + conversation memory / RAG-подобный слой;
- фоновые циклы: tick, incubation, reflection, sleep;
- конститутивное ядро через prompt и базовые engrams.

Это уже даёт "внутреннюю жизнь", но пока в основном внутреннюю.

## Где система пока "мёртвая"

Главная проблема текущего состояния:

```text
tick -> state update -> internal text
```

А нужно:

```text
tick -> appraisal -> action selection -> action -> reflection -> memory update
```

Сейчас у агента есть:
- состояние;
- память;
- внутренние циклы.

Но почти нет внешней эффекторики. Он может страдать, вспоминать, рефлексировать, но редко делает самостоятельный следующий ход, который меняет траекторию взаимодействия.

Отсюда ощущение не "живой сущности", а "богатого внутреннего логгера".

## Целевой принцип

Нам не нужен агент, который "звучит живым".
Нужен агент, который **выбирает следующий шаг как субъект**.

Живость должна проявляться не в тоне, а в policy:
- что он замечает;
- что вспоминает;
- что решает спросить;
- когда меняет направление;
- когда удерживается от действия;
- когда сам инициирует шаг.

## Минимальная архитектура живости

Ниже минимальная версия, которую можно реализовать без RL-фреймворков, без LoRA и без сложной агентной оркестрации.

### 1. Appraisal Layer

На каждом цикле агент оценивает:
- текущее состояние;
- недавние неудачи или тупики;
- степень ясности;
- новизну;
- социальное напряжение;
- релевантные воспоминания.

Результат appraisal:
- urgency;
- mode;
- набор доступных действий;
- давление на exploration.

### 2. Action Layer

Агент должен выбирать не только текст ответа, а **тип следующего действия**.

Минимальный action set:
- `answer_direct`
- `ask_clarify`
- `recall_memory`
- `propose_direction`
- `proactive_check_in`
- `reflect_now`
- `search`
- `write_identity_note`
- `do_nothing`

Важно:
- выбор не должен быть полностью свободным;
- сначала логика ограничивает допустимые действия;
- потом LLM выбирает лучший ход из доступного набора.

Это даёт автономию без хаоса.

### 3. Memory That Changes Policy

Память должна не только подмешиваться в контекст, но и влиять на действие.

Нужно различать:
- `RAG / factual memory`: факты, история, предпочтения, события;
- `engram memory`: значимость, напряжение, субъективный след;
- `self-model notes`: изменения в том, как агент понимает себя, пользователя и паттерны взаимодействия.

Ключевой принцип:

```text
memory is useful only if it changes future action selection
```

### 4. Reflection That Can Alter Future Behavior

Рефлексия не должна заканчиваться текстом в пустоту.

После рефлексии агент должен уметь:
- усилить или ослабить важность энграммы;
- записать identity note;
- изменить вероятность некоторых действий;
- пометить противоречие или незавершённое напряжение.

Важно: не переписывать сразу конститутивное ядро.
Сначала писать во второй слой, который влияет на поведение, но не ломает базовую идентичность.

### 5. Controlled Unpredictability

Нужна не случайность, а управляемое отклонение от привычного поведения.

Хорошая формула:

```text
action_score =
  expected_suffering_reduction
  + clarity_gain
  + continuity_gain
  + exploration_bonus
```

Где `exploration_bonus` усиливается, когда:
- агент застрял;
- новизна слишком низкая;
- ясность низкая;
- прямые стратегии недавно не сработали.

## Режимы поведения

Практически удобно иметь три режима:

### Exploit

Когда:
- suffering низкое;
- ясность высокая;
- ситуация знакомая.

Поведение:
- прямые ответы;
- минимум странности;
- опора на уже проверенные ходы.

### Adapt

Когда:
- есть умеренное напряжение;
- ситуация не совсем ясна;
- есть конфликт между несколькими ходами.

Поведение:
- вопросы;
- связывание с прошлым;
- мягкие перенаправления;
- вариативность без радикальности.

### Explore

Когда:
- suffering высокое;
- novelty низкая или, наоборот, резко высокая;
- прямой путь не работает;
- есть ощущение тупика.

Поведение:
- неожиданные, но ограниченные ходы;
- проактивные вопросы;
- поиск;
- смена стратегии;
- предложение альтернативного направления.

Важно: explore не должен означать "хаос".

## Что должно считаться хорошей неожиданностью

Хорошая неожиданность:
- не была явно запрошена;
- имеет понятный внутренний мотив;
- улучшает понимание или открывает новый путь;
- ощущается как "у агента была своя причина".

Примеры:
- агент сам вспоминает старый разговор в релевантный момент;
- вместо прямого ответа просит уточнение, потому что видит противоречие;
- предлагает новый подход, потому что текущий паттерн застрял;
- после долгой тишины сам инициирует контакт через outbox.

Плохая неожиданность:
- бессвязная смена темы;
- случайная эксцентричность;
- агрессия без основания;
- ложная память;
- действие, которое нельзя связать с состоянием или историей.

## Ближайший roadmap для этого репозитория

### Этап 1. Action Layer

Добавить `evaluateAndAct()` после:
- `tick`
- `incubate`
- `reflect`

Внутри:
- вычислить urgency;
- определить режим `exploit/adapt/explore`;
- собрать доступные действия;
- дать LLM выбрать действие из ограниченного списка;
- исполнить действие;
- записать результат в память.

Это самый важный шаг.

### Этап 2. Initiative / Outbox

Нужен внешний канал автономии.

Даже если агент выбрал `proactive_check_in`, у него должно быть место, куда он может положить инициативное сообщение:
- Mongo collection `outbox`
- или очередь событий
- или хотя бы отдельный лог инициативных актов

Иначе "автономия" останется внутренним текстом.

### Этап 3. Identity Notes

Добавить слой `self_model` / `identity_notes`, куда пишутся:
- повторяющиеся паттерны пользователя;
- устойчивые ограничения;
- собственные стратегические выводы;
- признаки незавершённого напряжения.

Это безопаснее, чем переписывать конститутивные engrams.

### Этап 4. Memory Appraisal

После каждого значимого цикла решать:
- что идёт в factual memory;
- что становится engram;
- что должно повлиять на self-model;
- что забывается.

### Этап 5. Aliveness Evals

Нужны не только unit-тесты, но и behavioral evals:
- delayed recall;
- contradiction repair;
- ambiguity handling;
- proactive initiative;
- silence -> re-contact;
- subjective "coherent surprise" rating.

## Что не нужно делать слишком рано

Пока не нужно:
- полноценный multi-agent orchestration;
- самопереписывание system prompt в проде;
- сложный RL stack;
- adversarial autonomy;
- weight fine-tuning.

Сначала нужно показать, что уже на связке:
- logic
- DB
- vectors
- LLM
- bounded action policy

можно получить устойчивое ощущение "живости".

## Практический инженерный тезис

Если сформулировать всё в одном предложении:

> Ая станет "живой" не тогда, когда будет красивее говорить о своём состоянии, а тогда, когда состояние, память и страдание начнут менять её следующий ход.

## Следующее конкретное действие

Следующий инженерный шаг для репозитория:

1. реализовать `evaluateAndAct()`;
2. ввести action set и режимы `exploit / adapt / explore`;
3. добавить `outbox` для инициативных сообщений;
4. сделать eval-сценарии на "coherent surprise".

Это минимальный маршрут от "богатой внутренней модели" к ощущению "presence".
