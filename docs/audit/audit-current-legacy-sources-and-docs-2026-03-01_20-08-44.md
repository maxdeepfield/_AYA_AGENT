## 4) Аудит текущего проекта (`she-is-not-alone-g`)

### Что реализовано хорошо

1. `Kernel` с fast/work/deep ticks и живым runtime-контуром.
2. `DriveEngine` + инициатива (curiosity/closure/social/novelty).
3. `QuestionEngine` и явная работа с вопросами/ответами.
4. `MemoryHub` с эпизодами, retrieval и Mongo fallback-логикой.
5. `ToolRouter` (минимальный набор: `web_search`, `fetch_url`).
6. Proactive messaging и фоновое исследование.
7. WS gateway (`src/index-ws.ts`) + UI в `ui-expo`.
8. Базовый identity-guard (`SelfAnchor`).

### Что сделано слабо/частично

1. `SingleWriter` — скорее очередь персистенции коммитов, чем жесткий глобальный транзакционный арбитр состояния.
2. Тесты покрывают только часть поведения (ядро/память/долгие сценарии покрыты неполно).
3. Много legacy-следов по документации: фактическое состояние выше R0, но docs местами описывают прошлую фазу.
4. В `src` сохраняется hard-exit паттерн (`process.exit(...)`) в entrypoints и аварийных путях.

### Связь с legacy

1. От `she-is-not-alone-y` взято структурное разделение по подсистемам.
2. От `she-is-not-alone` — typed-модульность и попытка formal contracts.
3. От `she-exists` — ambition-слой (identity/homeostasis/proactivity/inner loop), но в более компактной форме.

## 5) Подробный аудит legacy-проектов

## `she-is-not-alone-y`

Реализовано:

1. Базовый runtime агента, Mongo repository, `single-writer`, chat API, reasoning/homeostasis контур.

Плохо сделано:

1. Часть write-путей обходит `SingleWriter` (прямые записи через repository).
2. Нет зрелого auth/security слоя для API/WS.
3. Местами placeholder-логика под видом полного механизма.
4. Проблемы кодировок (mojibake) в документах/комментариях.

Связь с текущим проектом:

1. Прямой предок структуры `Kernel/Drive/Question/Memory`.
2. Текущий код исправляет часть монолитности и усиливает типизацию/разделение.

## `she-is-not-alone`

Реализовано:

1. Крупный TS-каркас (kernel, memory, reasoning, tools, kill-switch, runner).

Плохо сделано:

1. Много `Math.random` в системных путях (недетерминированность поведения).
2. `TODO` в важных метриках reasoning.

## 7) Что критично для цели «со-трансцендентности», а что вторично

Критично:

1. Непрерывность совместного контекста (`joint attention`), не только внутренняя рефлексия агента.
2. Явная связка вопрос -> ответ -> совет -> измеримый эффект.
3. Прозрачный heartbeat: “что делаю”, “чего жду”, “что хочу спросить у пользователя”.
4. Контролируемая инструментальность: инструменты должны усиливать диалог, а не уводить агента в автономную “самозанятость”.

Вторично (на этом этапе):

1. Максимальная сложность self-evolution контуров.
2. Массовое расширение сенсоров без сценариев пользовательской ценности.
3. Попытка перенести AGI-эксперименты из legacy без фильтра.

## 8) План улучшений (без ломки текущего ядра)

### Фаза A — Нормализация и консолидация

1. Очистить docs от дубликатов/пустышек (`CHAT_MODE_FIX.md`, legacy copies как canonical sources).
2. Синхронизировать root docs с текущим фактом (WS/UI, tool router, RAG path, health checks).
3. Зафиксировать одну “правду” по моделям Ollama и runtime-профилю.

### Фаза B — Усиление качества реализации

1. Довести `SingleWriter` до реального арбитра инвариантов, а не только очереди персистенции.
2. Добавить интеграционные тесты: `Kernel + MemoryHub + ToolRouter + QuestionEngine`.
3. Снизить hard-exit зависимость (graceful failure path вместо `process.exit` где возможно).

### Фаза C — Со-трансцендентный runtime

1. Ввести explicit joint-attention workflow:
   - агент предлагает объект внимания (ссылка/наблюдение);
   - обсуждение фрагментов;
   - фиксация совместных выводов в память.
2. Сделать proactive-контур менее “монологичным”: приоритет вопросов высокой ценности пользователю.
3. Добавить метрики совместности:
   - доля действий с user-impact;
   - decision-lift-from-advice;
   - unresolved-shared-threads.

## 9) Главные риски, если оставить как есть

1. Док-дрейф: команда будет ориентироваться на устаревшие R0-описания.
2. Архитектурная иллюзия “single writer already done” при фактической недозавершенности инвариантов.
3. Повторение legacy-ошибки: рост сложности быстрее, чем рост воспроизводимости и тестируемости.
4. Потеря фокуса цели: из “co-intelligence с пользователем” в “автономный генератор активности”.
