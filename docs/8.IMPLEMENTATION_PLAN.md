# Implementation Plan

Этот документ превращает новую линию проекта в прямой порядок работ.
Не в теорию.
Не в ещё один манифест.

А в то, что можно реально делать по шагам.

## Общий Подход

Не переписывать Aya ещё раз.
Не пытаться очистить старую систему до идеала.

Сделать рядом новый минимальный прототип.
Aya оставить как архив наблюдений.

## Где Делать Новый Прототип

Не внутри старой сложной логики Aya.
Лучше в отдельной новой папке внутри репозитория.

Предлагаемая структура:

```text
aya-one/
  prototype/
    package.json
    tsconfig.json
    src/
      index.ts
      types.ts
      drives.ts
      objective.ts
      repeat-tracker.ts
      candidate-generation.ts
      candidate-selection.ts
      execute-step.ts
      runtime-loop.ts
      memory.ts
      logger.ts
      adapters/
        terminal.ts
        web.ts
```

Если захочется ещё чище, можно даже вынести это в соседний репозиторий.
Но на первый проход отдельной папки внутри текущего достаточно.

## Что Не Переносить Из Aya

В новый прототип не переносить автоматически:

- homeostasis как центр поведения
- anxiety / suffering / arousal / affection
- expectation-reality gap
- identity notes
- reflection loops
- proactive_check_in как базовый механизм
- goal layer в старом виде
- длинные prompt-манифесты про внутреннее состояние
- сложную autobiographical memory

Если что-то из этого окажется нужно потом, добавлять только после того, как минимальный прототип уже живёт без них.

## Что Переносить Из Aya Можно

Только то, что действительно полезно и не тащит старую философию:

- terminal adapter
- web adapter
- базовый логгер
- возможно, часть обвязки LLM вызова
- удобные утилиты для структурированного вывода

То есть переносить не поведение, а инфраструктуру.

## Очерёдность Реализации

### Шаг 1. Каркас проекта

Сделать пустой `prototype/`:

- package.json
- tsconfig.json
- src/index.ts
- минимальный запуск через `npm run dev` или `npm start`

Критерий завершения:

- проект собирается
- стартует пустой loop
- логирует один tick

### Шаг 2. Типы

Реализовать типы из `6.DATA_MODEL.md`:

- AgentSnapshot
- Objective
- DriveState
- StepCandidate
- StepRecord
- ConsequenceEntry
- RepeatTracker

Критерий завершения:

- типы существуют
- runtime может создать начальный snapshot

### Шаг 3. Начальное состояние

Сделать функцию инициализации:

- стартовые drives
- пустая память последствий
- `currentObjective = null`

Критерий завершения:

- система умеет стартовать без пользователя
- в логе виден первый snapshot

### Шаг 4. Objective creation

Сделать самый простой `ensureObjective()`:

- если objective нет, брать ближайший объект среды
- если есть пользовательское направление, создавать objective из него
- если objective насыщен, закрывать его и создавать следующий минимальный

Критерий завершения:

- у существа почти всегда есть одно текущее дело
- objective не размножается

### Шаг 5. Candidate generation

Сделать `proposeCandidates()`:

- 2-5 bounded steps
- только маленькие реальные шаги
- без длинных планов

Примерно такие шаги:

- read one file
- run one terminal command
- run one web query
- compare last two results
- write one short synthesis note
- wait

Критерий завершения:

- на каждый tick есть небольшой пул кандидатов

### Шаг 6. Candidate selection

Сделать простой scoring:

- driveSupport
- estimatedNovelty
- estimatedRisk
- repeatPenalty

Никакой сложной магии.

Критерий завершения:

- система может выбрать один шаг из нескольких
- выбор меняется после пустых повторов

### Шаг 7. Step execution

Сделать `executeStep()`:

- terminal
- web
- read
- wait

Пока без write, если не нужен.

Критерий завершения:

- один тик делает один шаг
- шаг даёт структурированный результат

### Шаг 8. Consequence memory

Сделать `applyStepConsequences()`:

- useful / empty / failed / blocked
- objective progress
- short consequence memory
- update repeat tracker

Это один из важнейших шагов.

Критерий завершения:

- пустые шаги реально становятся дорогими
- useful шаги реально сдвигают objective

### Шаг 9. Repeat suppression

Сделать отдельный repeat tracker:

- нормализованный fingerprint шага
- рост штрафа за повтор
- быстрое подавление одинаковых пустых шагов

Критерий завершения:

- система не читает один и тот же README двадцать раз подряд
- система не делает почти одинаковый web search десять раз подряд

### Шаг 10. Minimal synthesis

Сделать маленький `synthesize`-ход:

- что уже найдено
- что повторяется
- что теперь лучший следующий shift

Не reflection. Не исповедь.
Просто артефакт смены фазы.

Критерий завершения:

- после нескольких сборов подряд система умеет перейти к короткому синтезу

### Шаг 11. User interaction

Только после этого добавить внешний пользовательский слой.

Принцип:

- user gives direction
- system updates current objective
- tools remain private

Критерий завершения:

- пользователь не выбирает terminal/web напрямую
- система сама превращает направление в шаг

## Самый Первый Релиз

Первый живой релиз прототипа считается достаточным, если он умеет:

- стартовать с пустого состояния
- создать одно текущее дело
- сделать один bounded step
- запомнить результат
- не повторить пустой шаг слишком много раз
- сделать короткий synthesis после насыщения

Этого уже достаточно для первого настоящего эксперимента.

## Что Не Надо Делать Сразу

Не надо сразу добавлять:

- долгую биографическую память
- rich self-model
- эмоции
- философские объяснения identity
- сложную социальную динамику
- многослойный planner
- несколько concurrent objectives

Если новый прототип снова не оживёт без этого, значит проблема глубже, и эти слои только замаскируют её.

## Минимальные Тесты

### Тест 1

Objective есть -> система делает bounded step.

### Тест 2

Пустой шаг -> repeat penalty растёт.

### Тест 3

Несколько пустых похожих шагов -> система меняет тип действия.

### Тест 4

После нескольких сборов -> появляется synthesis, а не ещё один сбор.

### Тест 5

User direction меняет objective, но не превращает пользователя в selector tool'ов.

## Где Остановиться После Каждого Шага

Очень важно не снова уйти в бесконечную архитектуру.

После каждого крупного этапа надо останавливаться и смотреть:

- стало ли поведение интереснее
- стало ли меньше ритуала
- стало ли меньше шума
- не вернулась ли старая болезнь в новой форме

Если ответ нет, не добавлять новый слой автоматически.
Сначала признать, что текущий слой не сработал.

## Коротко

Порядок такой:

1. Новый пустой прототип рядом со старой Aya.
2. Типы.
3. Один objective.
4. Несколько drives.
5. Candidate generation.
6. Step selection.
7. Step execution.
8. Consequences.
9. Repeat suppression.
10. Minimal synthesis.
11. Только потом пользовательский слой.

## Главный Принцип

Новый проект нужно строить не как улучшенную драму Aya.
А как минимальную машину предпочтений, последствий и выбора.

Если это заработает, всё остальное можно будет наращивать потом.
Если не заработает, значит не надо было тащить в него ещё больше психологии.
